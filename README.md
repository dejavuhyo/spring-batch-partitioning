# Spring Batch Partitioning

## 1. 설명
Spring Batch의 Partitioning(파티셔닝)은 대량의 데이터를 처리할 때 성능을 극대화하기 위한 기능dl다. 하나의 큰 작업을 여러 개의 작은 작업 단위로 쪼개어 멀티 쓰레드(Multi-threaded) 환경에서 병렬로 실행하는 기법이다.

## 2. 개발환경

* OpenJDK 21

* spring-boot 3.4.0

* mybatis-spring-boot-starter 3.0.4

* lombok 1.18.38

* PostgerSQL 17.9


## 3. 전체 구조

* Job: 배치 전체 실행 단위

* Manager Step: 데이터를 어떻게 나눌지 결정하는 역할

* Worker Step: 실제로 데이터를 읽고 처리하는 역할

> Manager가 일을 쪼개고, Worker들이 각각 맡아서 처리하는 구조이다.

## 4. 파티셔닝 사용 이유
단순한 멀티 쓰레드 모델보다 파티셔닝이 유리한 이유는 다음과 같다.

### 1) Thread-Safe 걱정 감소: 각 Worker Step이 별도의 ItemReader 인프라를 가지므로, 쓰레드 세이프하지 않은 Reader도 사용할 수 있다.

* 독립적인 상태 관리: 특정 파티션에서 오류가 나면 해당 파티션만 재시작하면 된다.

* 확장성(Scalability): 단순히 한 컴퓨터 내의 쓰레드를 늘리는 것을 넘어, 원격 서버(Remote Partitioning)로 작업을 분산할 수도 있다.

## 5. 역할
이 구조에서는 보통 다음과 같은 역할이 나뉜다.

* Partitioner: 데이터를 여러 파티션으로 나눈다.

* Manager Step: Partitioner가 만든 파티션들을 받아서, 각 파티션별로 Worker Step을 실행시킨다.

* Worker Step: 실제 데이터 처리를 한다.

## 6. 이 구조의 장점

### 1) 처리 속도 향상
여러 파티션을 병렬 처리하므로 전체 시간이 줄어든다.

### 2) 대용량 데이터에 적합
데이터가 많을수록 효과가 크다.

### 3) 범위 기반 처리에 유리
ID 범위, 날짜 범위, 구간별 데이터 처리에 잘 맞다.

### 4) 장애 영향 범위 축소
한 파티션에 문제가 생겨도 다른 파티션과 분리되어 다루기 쉽다.

## 7. 주의할 점
파티셔닝이 무조건 좋은건 아니다. 아래를 주의해야 한다.

### 1) 파티션 수를 너무 많이 잡지 않기
너무 잘게 쪼개면 오히려 오버헤드가 커진다.

### 2) DB 부하
동시에 여러 스레드가 DB를 읽고 쓰기 때문에 DB에 부담이 갈 수 있다.

### 3) 범위 분할 기준이 중요
ID 기준이 가장 단순하지만, 데이터 분포가 한쪽으로 치우치면 어떤 파티션은 너무 많고 어떤 파티션은 너무 적을 수 있다.

### 4) 중복/누락 방지
파티션 범위가 겹치면 중복 처리되고, 빠지면 누락된다. 그래서 Partitioner 구현이 매우 중요하다.
